# Архітектура та детальний опис проєкту "SkillForge"

## 1. Загальний огляд (Executive Summary)
**SkillForge** — це масштабована, розподілена платформа для онлайн-навчання (LMS - Learning Management System), спроєктована на основі мікросервісної архітектури. Платформа створена для задоволення потреб B2B та B2C сегментів завдяки підтримці мульти-організаційної моделі (Multi-tenancy). 

Мета проєкту: надати інструмент для гнучкого управління навчальними програмами, оцінюванням студентів та розмежуванням доступу між різними групами викладачів та адміністраторів.

## 2. Ролеві моделі та розмежування прав (RBAC & Multi-tenancy)
Платформа базується на складній ієрархічній системі ролей:
- **SUPER_ADMIN (Глобальний Куратор)**: Має доступ до всієї системи. Може створювати нові організації (навчальні заклади/компанії) та призначати адміністраторів для цих організацій.
- **ORG_ADMIN (Адміністратор організації)**: Куратор, який має повні права в межах своєї організації. Може додавати викладачів, студентів та керувати контентом своєї організації.
- **STANDARD CURATOR (Куратор)**: Відповідає за створення навчальних програм (курсів) всередині призначеної йому організації.
- **EDUCATOR (Викладач)**: Отримує доступ лише до тих курсів, у яких він призначений викладачем в межах організації. Може додавати слухачів на курс та перевіряти їхній прогрес.
- **LEARNER (Студент/Слухач)**: Споживач контенту. Може записуватися на загальнодоступні курси (Enrollment), проходити уроки та здавати тести.

## 3. Мікросервісна Архітектура (Microservices Topology)
Проєкт розділено на 4 автономні сервіси, які взаємодіють між собою через REST API:

### 3.1. API Gateway (`api-gateway` | порт 8080)
- **Технології**: Spring Cloud Gateway (WebFlux).
- **Роль**: Єдина точка входу (Single Point of Entry) для всіх запитів з фронтенду.
- **Функціонал**:
  - Обробка CORS політик (`allowCredentials: true`).
  - Маршрутизація запитів до відповідних сервісів на основі URL (наприклад, `/api/users/**` -> `user-service`).
  - Видалення внутрішніх заголовків (наприклад, `RemoveRequestHeader=Origin`), щоб уникнути конфліктів у downstream сервісах.

### 3.2. User Service (`user-service` | порт 8082)
- **Технології**: Spring Boot, Spring Security, JWT, Spring Data JPA, H2.
- **Роль**: Управління ідентифікацією (Identity Management) та організаціями.
- **Функціонал**:
  - Генерація та валідація JWT-токенів (HmacSHA256).
  - Управління користувачами (`User`).
  - Управління організаціями (`Organization`) та членством (`OrganizationMember`).
  - Ендпоінт `/api/me/context` для швидкого надання інформації про те, до яких організацій належить користувач.

### 3.3. Course Service (`course-service` | порт 8083)
- **Технології**: Spring Boot, Spring Security (Resource Server), Spring Data JPA, H2.
- **Роль**: Основний бізнес-домен (Core Business Logic).
- **Функціонал**:
  - Управління структурою: `Course` -> `CourseModule` -> `Lesson` -> `Quiz`.
  - Управління записами: `Enrollment` (зв'язок між користувачем і курсом).
  - Валідація JWT токенів та перевірка прав доступу користувача до курсу на основі `organizationId`.
  - Завантаження студентів (Learners) викладачем.

### 3.4. Core Service (`core-service` | порт 8081)
- **Роль**: Заділ під загальні інтеграції, обробку медіафайлів (відео/фото), відправку email-повідомлень та іншу асинхронну логіку в майбутньому.

## 4. Схема Бази Даних (Domain Model)

### 4.1. Домен Користувачів (`user-service`)
- `users`: id, email, password_hash, first_name, last_name, role, curator_global_role.
- `organizations`: id, name, created_at.
- `organization_members`: id, organization_id, user_id, member_kind, curator_org_role. (Забезпечує зв'язок Many-to-Many між користувачами та організаціями).

### 4.2. Домен Курсів (`course-service`)
- `courses`: id, organization_id, title, description, status (DRAFT, PUBLISHED).
- `course_modules`: id, course_id, name, sort_order.
- `lessons`: id, module_id, title, content.
- `quizzes`: id, lesson_id, title, passing_score.
- `questions`: id, quiz_id, text, type (SINGLE_CHOICE, MULTIPLE_CHOICE).
- `answer_options`: id, question_id, text, is_correct.
- `enrollments`: id, course_id, user_id, enrolled_at.

*Оптимізація Hibernate*: Використовується `default_batch_fetch_size: 16` для запобігання проблеми "N+1" та `MultipleBagFetchException` під час завантаження вкладених структур (курсів і модулів).

## 5. Безпека та JWT Взаємодія (Security Flow)
1. **Логін**: Користувач відправляє email та пароль на `/auth/login` (User Service).
2. **Токен**: Якщо дані правильні, система генерує JWT токен, який містить `userId`, `role` та `curatorGlobalRole`. Токен підписується спільним таємним ключем (`jwt.secret`).
3. **Авторизація**: Фронтенд зберігає токен у `localStorage` та додає його в заголовок `Authorization: Bearer <token>` до всіх наступних запитів.
4. **Міжсервісна перевірка**: Коли `course-service` отримує запит, його внутрішній `JwtAuthenticationFilter` перевіряє підпис токена.
5. **Синхронізація (Context Fetching)**: `CourseApplicationService` робить синхронний виклик до `user-service` через `UserDirectoryClient` для перевірки доступу користувача до конкретної організації перед виконанням мутацій (створення чи редагування курсу).

## 6. Frontend Архітектура
Фронтенд побудований як Single Page Application (SPA) з акцентом на сучасний UX:
- **React 19 + TypeScript**: Забезпечують сувору типізацію (interfaces для API відповідей) та швидкодію.
- **Material UI (MUI) v6**: Використовується як дизайн-система для побудови компонентів (кнопки, таблиці, картки, діалоги).
- **Framer Motion**: Використовується для анімації появи елементів на екрані (наприклад, сторінка логіну).
- **React Router**: Забезпечує Protected Routes (захищені маршрути). Користувач перенаправляється на сторінку `/forbidden`, якщо намагається відкрити маршрут, який не відповідає його ролі (наприклад, Educator намагається відкрити `/curator/organizations`).
- **Глобальний Стейт**: Використовується React Context API (`AuthContext`), який тримає `meContext` (дані про організації) і робить їх доступними в будь-якому компоненті.

## 7. Автоматизація та CI/CD
- **Локальний запуск**: Здійснюється через мультипроцесний `run_script.bat`, який паралельно запускає всі Spring Boot застосунки та React фронтенд, а також акуратно їх завершує (`taskkill`) при закритті.
- **Continuous Integration (GitHub Actions)**: Налаштовано `.github/workflows/ci.yml` для автоматичної збірки Maven та npm проєктів при кожному `git push` в гілку `main`. Це гарантує, що код завжди залишається в робочому (компільованому) стані.

## 8. Перспективи розвитку (Future Work)
- **Асинхронні події**: Заміна синхронних REST викликів між `course-service` та `user-service` на черги повідомлень (RabbitMQ або Apache Kafka) для забезпечення Event-Driven Architecture.
- **Кешування**: Використання Redis для кешування результатів `/api/me/context`, щоб зменшити навантаження на базу даних.
- **Аналітика**: Створення окремого сервісу звітності (Reporting Service) для візуалізації прогресу студентів у вигляді графіків.
