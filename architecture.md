# Архітектура: автоматизована система управління підвищенням кваліфікації ІТ-працівників

Цей документ фіксує цільову архітектуру монорепозиторію та служить **контекстом для подальшої розробки** (фронтенд, бекенд, інтеграції, безпека).

---

## 1. Мета системи

Система підтримує **організаційний цикл підвищення кваліфікації** ІТ-працівників: планування навчальних програм, призначення викладачів і груп, проходження курсів, виконання завдань і перевірка результатів. Користувачі мають різні повноваження залежно від ролі.

---

## 2. Монорепозиторій (високорівнева структура)

| Шар | Технології | Призначення |
|-----|------------|-------------|
| **Frontend** | React, TypeScript | SPA: кабінети Curator / Educator / Learner, форми, перегляд курсів і прогресу |
| **API Gateway** | Spring Cloud Gateway | Єдина точка входу для клієнта; маршрутизація до мікросервісів; CORS; (за потреби) агрегація простих відповідей |
| **Service Discovery** | Netflix Eureka | Реєстрація інстансів сервісів; резолв імен типу `http://user-service/...` замість жорстких URL |
| **User Service** | Spring Boot | Користувачі, автентифікація/авторизація (або інтеграція з IdP), ролі, профілі |
| **Course Service** | Spring Boot | Курси, модулі/уроки, групи, призначення, завдання, здачі, статуси проходження |

*Примітка:* у процесі імплементації можуть з’явитися додаткові модулі (наприклад, спільна бібліотека DTO, конфігурація, Docker Compose). Цей документ описує **логічну** межу відповідальності; фізичні папки в репозиторії можуть уточнюватися в README або в окремих технічних нотатках.

## 3. Ролі та повноваження

### 3.1. Curator (куратор навчання)

- Створює та редагує **курси** (структура, матеріали, терміни за потреби).
- **Призначає викладачів (Educator)** на курси або групи.
- Може переглядати загальну картину по програмах (залежить від реалізації звітності).

**Типові дії:** CRUD курсів, призначення менторів, публікація/архівація курсу.

### 3.2. Educator (викладач / ментор)

- **Менторить призначену групу** (або потік) по конкретному курсу.
- **Перевіряє завдання** учнів: приймає, відхиляє з коментарем, виставляє оцінку/статус (за моделлю предметної області).
- Може бачити список учнів у своїх групах і їх прогрес.

**Типові дії:** перегляд здач, фідбек, зміна статусу перевірки.

### 3.3. Learner (слухач / ІТ-працівник)

- **Проходить курси**, до яких отримав доступ.
- Виконує **завдання**, завантажує результати (якщо передбачено).
- Бачить **власний прогрес** і коментарі викладача.

**Типові дії:** перегляд контенту, відправка робіт, перегляд оцінок/статусів.

### 3.4. Матриця доступу (орієнтир)

| Дія | Curator | Educator | Learner |
|-----|:-------:|:--------:|:-------:|
| Управління курсами (створення/редагування) | ✓ | — | — |
| Призначення викладачів | ✓ | — | — |
| Перевірка завдань своїх груп | — | ✓ | — |
| Проходження курсу / здача завдань | — | — | ✓ |

Деталі ендпоінтів і політик безпеки задаються в User Service (ролі, JWT claims) та перевіряються в Course Service (доступ до ресурсу).

## 4. Мікросервіси та відповідальність

### 4.1. API Gateway

- Приймає всі HTTP-запити від фронтенду (типово префікси на кшталт `/api/users/**`, `/api/courses/**`).
- Маршрутизує на **User Service** або **Course Service** через **Eureka** (`lb://user-service`, `lb://course-service`).
- Узгоджує **CORS** для dev (localhost) та прод-оригінів.
- Може містити **глобальні фільтри**: логування correlation ID, rate limiting (опційно), базова валідація заголовків.

### 4.2. Eureka Server

- Окремий процес реєстрації сервісів.
- **User Service** і **Course Service** реєструються під стабільними іменами додатків (`spring.application.name`).
- Gateway отримує актуальні інстанси для балансування навантаження (за наявності кількох реплік).

### 4.3. User Service

**Відповідає за:**

- Облікові записи: реєстрація (якщо потрібно), профіль, зв’язок з корпоративним ідентифікатором (опційно).
- **Автентифікація:** видача JWT (або інтеграція з OAuth2/OIDC — за вибором імплементації).
- **Авторизація на рівні ролей:** Curator / Educator / Learner; можливі додаткові атрибути (наприклад, `userId` для зв’язку з Course Service).
- Ендпоінти для фронтенду: логін, `/me`, зміна профілю, адміністративні операції для куратора (наприклад, призначення ролі — якщо це в межах системи).

**Межа даних:** користувачі, ролі, облікові дані; **не** дублювати повну структуру курсів (лише ідентифікатори користувачів для посилань).

### 4.4. Course Service

**Відповідає за:**

- **Курси:** метадані, видимість, стан (чернетка, опублікований, архів).
- **Навчальний контент:** модулі, уроки, матеріали (URL, файли — залежить від реалізації сховища).
- **Групи / потоки** навчання та **призначення** Educator і Learner (хто в якій групі по якому курсу).
- **Завдання** та **здачі (submissions):** статуси, дедлайни, файли/текст, історія перевірки.
- **Бізнес-правила:** лише Educator з доступом до групи може перевіряти відповідні здачі; Learner бачить лише власні дані; Curator керує курсами та призначеннями викладачів.

**Межа даних:** усе, що стосується навчального процесу; посилання на користувачів — через `userId` (з User Service), без зберігання паролів.

## 5. Взаємодія між сервісами

### 5.1. Потік запиту від клієнта

1. Браузер → **Gateway** (з токеном у заголовку `Authorization`, якщо використовується JWT).
2. Gateway → **User Service** (наприклад, логін, оновлення токена, профіль) або → **Course Service** (курси, завдання, здачі).
3. Ім’я хоста сервісу резолвиться через **Eureka** (`lb://...`).

### 5.2. Синхронна взаємодія User ↔ Course

- **Course Service** при перевірці доступу може покладатися на **claims у JWT**, які видав **User Service** (роль, `sub`/`userId`), без звернення назад при кожному запиті (stateless).
- За потреби **валідація токена** або **отримання актуальної ролі** може робитися через виклик User Service (рідше, для складніших сценаріїв).
- Опційно: події домену (наприклад, “користувач деактивований”) — через чергу в майбутньому; на старті достатньо синхронних API та узгоджених ідентифікаторів.

### 5.3. Типові сценарії за ролями (логіка розподілу)

| Сценарій | Сервіс(и) |
|----------|-----------|
| Вхід, отримання профілю | User Service (через Gateway) |
| Curator створює курс | Course Service |
| Curator призначає Educator на курс/групу | Course Service (запис зв’язку `educatorId`–`courseId`/`groupId`) |
| Learner переглядає курс і здає завдання | Course Service |
| Educator перевіряє здачу | Course Service |
| Список користувачів для адмін-UI куратора | User Service (+ Course Service для контексту груп) |

## 6. Дані та узгодження ідентифікаторів

- Первинний ідентифікатор користувача **`userId`** (UUID або Long) узгоджується між сервісами.
- У **Course Service** таблиці зберігають `userId` для Learner, Educator, автора курсу (Curator), без дублювання ПІБ з User Service у критичних місцях (для відображення фронт може викликати User Service або отримувати агреговані DTO — за рішенням імплементації).

---

## 7. Безпека (орієнтир для імплементації)

- **HTTPS** у продакшені; у dev — HTTP на localhost.
- **JWT** (або сесії) видає User Service; Gateway може пропускати токен прозоро або валідувати підпис (залежить від обраної схеми).
- **RBAC:** перевірка ролі на Gateway (опційно) і **обов’язково** в сервісах на рівні бізнес-операцій.
- **Course Service** не довіряє лише клієнту: перевіряє, що `userId` з токена відповідає ресурсу (наприклад, learner здає лише від свого імені).

---

## 8. Спостережуваність і якість (рекомендовано для роботи)

- **Actuator** + health checks на кожному сервісі.
- Узгоджений **correlation ID** у заголовках для трасування запиту через Gateway → сервіс.
- Централізовані логи (опційно в рамках диплому/курсової): файли або ELK-подібний стек — за часу та вимог керівника.

---

## 9. Еволюція архітектури

Можливі наступні кроки без зламу концепції:

- Окремий **Notification Service** (email/чат про дедлайни і перевірки).
- **Файлове сховище** (S3/minio) для матеріалів і здач.
- **Message broker** для асинхронних подій (курс опубліковано, здача на перевірці).
- **Config Server** для централізованої конфігурації середовищ.
